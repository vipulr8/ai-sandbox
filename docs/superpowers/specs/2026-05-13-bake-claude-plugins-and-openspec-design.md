# Bake Claude plugins + OpenSpec — design

**Date:** 2026-05-13
**Status:** Pending implementation plan

## Goal

Make every project that launches the ai-sandbox container start with a curated set of Claude Code plugins and developer tools pre-installed, so the user doesn't have to run any install command before getting work done.

Concretely:

| Tool | Today | After this change |
|------|-------|-------------------|
| `superpowers` plugin (Claude Code marketplace) | User must `/plugin install superpowers@claude-plugins-official` per project, install is then trapped inside the per-project `~/.claude/` bind-mount | Baked into image; loaded automatically by a `claude` wrapper that injects `--plugin-dir` flags. User's own `/plugin install foo` continues to work normally and persists per-project. |
| `openspec` CLI (`@fission-ai/openspec` npm) | Not installed | Installed globally at build time. `openspec init` per project is the user's call. |

Both tools become part of the image's "batteries-included" surface, joining the VS Code extensions, the Claude CLI, the language toolchains, and `gitleaks` already there.

## Non-goals

- **Auto-init OpenSpec in each project.** `openspec init` writes opinionated slash-command files into the project; we don't make that choice for the user.
- **Per-launch plugin enable/disable flags.** Plugins listed in `claude-plugins.txt` are always loaded. To customize, edit the file and rebuild — same as `vscode-extensions.txt`.
- **Auto-update of baked plugins.** Rebuilding the image is how you upgrade. Matches the rest of the image's update model (`DISABLE_AUTOUPDATER=1` is set throughout).
- **Marketplace-style `enabledPlugins` declaration via managed-settings.** The wrapper handles loading; declaring it twice would be redundant and would risk Claude trying to network-fetch a plugin that's already locally available.
- **Multiple-marketplace support beyond what `claude-plugins.txt` lists.** Each line is `name url [ref]`. Users wanting a different marketplace at runtime still use `/plugin marketplace add` themselves.

## Architecture

Five additions, no removals.

### 1. `claude-plugins.txt` — list-file source of truth

Repo-root file, format mirroring `vscode-extensions.txt`. One plugin per line:

```
# Format: <name> <git-url> [<ref-or-sha>]
# Empty lines and # comments are ignored.
superpowers https://github.com/obra/superpowers.git f2cbfbefebbfef77321e4c9abc9e949826bea9d7
```

The sha is taken from the official marketplace catalog (`~/.claude/plugins/marketplaces/claude-plugins-official/.claude-plugin/marketplace.json`) at the time of writing, which gives reproducible builds and pins us to the same revision Claude's own marketplace would install. If the third column is omitted, `main` is checked out.

### 2. `scripts/install-claude-plugins.sh` — build-time installer

Invoked from the Dockerfile. For each non-comment line:

1. `git clone --depth 1 <url> /opt/ai-sandbox/plugins/<name>/`
2. If `<ref>` provided:
   ```bash
   git -C /opt/ai-sandbox/plugins/<name> fetch --depth 1 origin <ref>
   git -C /opt/ai-sandbox/plugins/<name> checkout <ref>
   ```
3. `git -C /opt/ai-sandbox/plugins/<name> rev-parse HEAD > /opt/ai-sandbox/plugins/<name>/.installed-sha`
4. Verify `/opt/ai-sandbox/plugins/<name>/.claude-plugin/plugin.json` exists — if not, the source repo isn't a valid Claude plugin, fail the build with a clear message.

After all plugins are cloned, the script writes `/usr/local/bin/claude` (the wrapper, see below) and `chmod +x` it.

### 3. `/usr/local/bin/claude` — generated wrapper

The wrapper is generated at the END of `install-claude-plugins.sh`, after all plugins are in place. It captures the real `claude` binary's absolute path at generation time (resolved via `command -v claude` **before** `/usr/local/bin/claude` exists) and emits:

```sh
#!/bin/sh
# Generated by install-claude-plugins.sh. Do not edit; rebuild the image.
exec <real-claude-path> \
    --plugin-dir /opt/ai-sandbox/plugins/superpowers \
    "$@"
```

One `--plugin-dir` line per baked plugin, in `claude-plugins.txt` order.

PATH ordering: `/usr/local/bin` precedes `/usr/bin` and Node global-install paths on Debian-based images, so this wrapper transparently shadows the npm-installed `claude` binary for every caller — interactive shells, the VS Code extension, scripts, all of them. The real binary remains accessible at its original path for anyone who needs to bypass the wrapper.

### 4. Dockerfile — one new layer

Inserted right after Layer 7 (Claude CLI install) and before the container-hooks layer:

```dockerfile
# ── Layer 7.5: Claude plugins + OpenSpec ─────────────────────────
RUN npm install -g @fission-ai/openspec@latest && npm cache clean --force
COPY claude-plugins.txt /tmp/claude-plugins.txt
COPY scripts/install-claude-plugins.sh /tmp/install-claude-plugins.sh
RUN /tmp/install-claude-plugins.sh /tmp/claude-plugins.txt \
    && rm /tmp/install-claude-plugins.sh /tmp/claude-plugins.txt
```

Why Layer 7.5: the slow apt/Go/Rust/Node toolchain layers are above (cache-stable); the fast-changing plugins layer goes below so editing `claude-plugins.txt` doesn't invalidate any expensive layers. OpenSpec install is in the same `RUN` as a sibling concern — both are "developer tooling baked at build" and they share the network-fetch cost.

### 5. Documentation updates

**`CLAUDE.md`** — add a short subsection under "Common commands" or "Architecture" describing:
- Plugins listed in `claude-plugins.txt` are baked at `/opt/ai-sandbox/plugins/<name>/` and loaded via `--plugin-dir` flags injected by `/usr/local/bin/claude`.
- User-installed plugins via `/plugin install foo` continue to write to `~/.claude/plugins/` and are loaded by Claude's default scan — both layers coexist.
- OpenSpec is available globally as `openspec`. Users run `openspec init` per project themselves.
- To add a baked plugin: append a line to `claude-plugins.txt` and rebuild.

**`README.md`** — surface the baked tools in the user-facing feature list:
- Mention `superpowers` and `openspec init` are pre-installed.
- Note the `claude-plugins.txt` extension point for power users.

## Risks and tradeoffs

| Risk | Mitigation |
|------|------------|
| The real `claude` binary path changes between Claude CLI versions, breaking the wrapper | The install script resolves the path at build time via `command -v claude` and bakes the absolute path into the wrapper — so each image build is internally consistent. A `claude --claude-version` rebuild regenerates the wrapper. |
| Some other layer or future change puts a `claude` binary in `/usr/local/bin` and conflicts with our wrapper | The wrapper is the LAST writer to `/usr/local/bin/claude` in the build; any future layer would have to deliberately overwrite it. Detectable in code review. |
| Cloned-from-GitHub plugin code is now baked into every container | Same trust model as the rest of the image: we trust the marketplace's pinned sha, just like we trust the `npm install` of `@anthropic-ai/claude-code` itself. Pinning to a sha (column 3 of `claude-plugins.txt`) is the recommended practice and what the initial entry uses. |
| `--plugin-dir` is documented as a "development and testing" flag — could change behavior | It's a public, documented CLI flag. The docs explicitly support stacking it multiple times and describe its precedence interaction with installed plugins. If Anthropic deprecates it, we'd need to migrate — but this is the official escape hatch for the exact use case we have. |
| Plugin clones add to image size | `--depth 1` keeps clones lean. Each plugin is typically <5 MB. Adds maybe ~20 MB to a ~3 GB image. |
| OpenSpec's npm install adds a network dependency to the build | Already true for every other npm/apt/cargo install in the Dockerfile. No new failure mode. |

## Testing

No test suite — verification is empirical, matching the existing pattern.

1. **`shellcheck`** clean on `scripts/install-claude-plugins.sh`.
2. **Rebuild:** `./run.sh --build` succeeds. Look for build-log lines from `install-claude-plugins.sh` confirming the clone and the wrapper generation.
3. **Plugin presence at the baked path:**
   ```bash
   docker run --rm ai-sandbox:latest ls /opt/ai-sandbox/plugins/superpowers/.claude-plugin/plugin.json
   docker run --rm ai-sandbox:latest cat /opt/ai-sandbox/plugins/superpowers/.installed-sha
   ```
   Expected: plugin.json exists; `.installed-sha` matches the sha in `claude-plugins.txt`.
4. **OpenSpec installed:**
   ```bash
   docker run --rm ai-sandbox:latest openspec --version
   ```
   Expected: prints a version string, no errors.
5. **Wrapper shadows real binary:**
   ```bash
   docker run --rm ai-sandbox:latest which claude
   docker run --rm ai-sandbox:latest head -10 /usr/local/bin/claude
   ```
   Expected: `which claude` prints `/usr/local/bin/claude`; the file is a small shell script containing `--plugin-dir /opt/ai-sandbox/plugins/superpowers`.
6. **Real `claude` still launches under the wrapper:**
   ```bash
   docker run --rm ai-sandbox:latest claude --version
   ```
   Expected: prints the Claude CLI version (proves `exec` to the real binary works).
7. **Plugin actually loads (interactive gate — user runs this):**
   ```bash
   ./run.sh ~/some-project
   # Inside the container, run `claude` and at the prompt try a slash command from superpowers:
   /superpowers:brainstorming
   ```
   Expected: the slash command resolves (the brainstorming skill activates).
8. **Coexistence with user-installed plugins (interactive gate):**
   Inside the container, run `/plugin install <some-other-plugin>@claude-plugins-official`. Confirm it installs to `~/.claude/plugins/` (per-project state dir on host) and that `/plugin list` shows BOTH the baked superpowers AND the newly-installed plugin.
9. **shellcheck** on the generated wrapper:
   ```bash
   docker run --rm ai-sandbox:latest shellcheck /usr/local/bin/claude
   ```
   Expected: clean (it's a trivial `exec` script).

## Security

The baked plugin code runs as user `coder` (UID 1000) inside the container, the same trust boundary as Claude Code itself. Same model as the VS Code extensions already baked into the image: we vet what goes into the list file (`claude-plugins.txt`), pin to a specific sha for reproducibility, and rebuild to upgrade.

The wrapper is a 5-line shell script with no user-controlled inputs other than the `"$@"` it passes through unchanged. There is no injection surface: the `--plugin-dir` paths are hard-coded into the generated file at build time from values in `claude-plugins.txt`, which is a repo-root file under normal code review.

Container-enforced settings (the managed-settings work just landed in the previous branch) continue to apply: `PreToolUse` hooks still block remote-publishing for AI Bash calls regardless of which plugins are loaded, because hooks are enforced by Claude's settings loader, not by plugin code.
